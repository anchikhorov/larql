# Accuracy baseline — Gemma 3 4B Q4K, four engines (2026-05-17)

First-ever multi-engine accuracy capture using `larql accuracy`. Establishes
a regression baseline for K/V engine correctness: future runs should not
regress these numbers without an accompanying explanation in the engine
implementation.

## Reproducer

```sh
./target/release/larql accuracy \
  /Users/christopherhay/.cache/huggingface/hub/models--google--gemma-3-4b-it/snapshots/093f9f388b31de276ce2de164bdc2081324b9767 \
  --output-file crates/larql-kv/baselines/gemma3-4b-2026-05-17.json
```

Default engines (`standard,markov-rs,unlimited-context,turbo-quant`), default
corpora (100 parametric + 5 needles ≤8192 tokens + 20 conflict = 125 prompts
per engine, 500 total scored prompts). M3 Max, CPU only, 8-thread rayon.

## Headline table

```
Strategy             Param %  Param bits   InCtx %  InCtx bits   Follow %  Fallback %
markov-rs              69.0%        2.25      0.0%       15.50      25.0%       35.0%
standard               69.0%        2.25      0.0%       15.50      25.0%       35.0%
turbo-quant            69.0%        2.25      0.0%       15.50      25.0%       35.0%
unlimited-context      69.0%        2.21      0.0%       22.22      25.0%       30.0%
```

| Column | What it measures |
|---|---|
| Param % | Top-1 match rate on 100 parametric prompts (cued completion of facts the weights know). |
| Param bits | Mean `-log2 P(expected_first_token \| prompt)`. Lower = more confident. |
| InCtx % | Top-1 match rate on 5 needle-in-haystack prompts (planted answer in 512–8192-token context). |
| InCtx bits | Mean bits on the needle answer. Rises when the engine loses the K/V state holding the needle. |
| Follow % | Conflict corpus: % of 20 prompts where the engine followed the in-context override. |
| Fallback % | Conflict corpus: % where the engine ignored the override and returned the parametric answer. |

The remaining (100 − Follow − Fallback)% in the conflict column is "other" —
the model produced neither the override nor the parametric answer, typically
because it autocompleted a different sentence form (e.g. "Pablo" instead of
"Picasso" for the Mona Lisa prompt — Gemma autocompletes "Pablo Picasso").

## Wall-clock per engine

| Engine | Parametric | In-context | Conflict | Total |
|---|---:|---:|---:|---:|
| standard | 58.2s | 337.7s | 12.5s | **6.8 min** |
| markov-rs | 63.8s | 330.0s | 13.1s | 6.8 min |
| turbo-quant | 66.9s | 354.5s | 18.7s | 7.3 min |
| **unlimited-context** | 122.8s | **1896.2s** | 55.8s | **34.6 min** |

`unlimited-context` runs at default `window=512` — anything past the first
window forces a checkpoint-write + sub-prompt re-prefill, so the
8192-token needle alone runs 16 window-close cycles. The cost is
the design's price for unbounded-context support, not a bug.

## Key findings

### 1. Parametric correctness is bit-exact across all four engines

Same 69/100 top-1, identical 2.25 mean bits. The K/V-engine contract holds:
no engine's cache strategy alters the argmax (or even the probability mass)
on weight-level recall. This is the gate every K/V engine must pass before
its in-context behaviour is meaningful.

The `unlimited-context` parametric bits (2.21) is 0.04 lower than the other
three — within float noise; the engine's `process_q4k` path uses slightly
different intermediate ordering (`extend_current_q4k` → per-token forward)
than Standard's coarse-prefill batch. Not a divergence worth chasing.

### 2. Unlimited-context loses the needle at the very first window boundary

Per-prompt bits on the 5 needle lengths (lower is better; bits ≈ 0 means
the model is essentially certain about the next token):

| Context | standard | markov-rs | unlimited-context | turbo-quant |
|---|---:|---:|---:|---:|
| 512 tok | 0.00 (` AUR`) | 0.00 (` AUR`) | **0.00 (` AUR`)** | 0.00 (` AUR`) |
| 1024 tok | 0.01 (` AUR`) | 0.01 (` AUR`) | **25.46 (` The`)** | 0.01 (` AUR`) |
| 2048 tok | 22.71 (`\n\n`) | 22.71 (`\n\n`) | 27.03 (` The`) | 22.71 (`\n\n`) |
| 4096 tok | 24.64 (`\n\n`) | 24.64 (`\n\n`) | 29.96 (` The`) | 24.64 (`\n\n`) |
| 8192 tok | 30.12 (` The`) | 30.12 (` The`) | **28.66 (` The`)** | 30.12 (` The`) |

Two observations:

- **Standard / markov-rs / turbo-quant are bit-identical** on needles. The
  4-bit Lloyd-Max codec preserves enough K/V signal that the needle
  retrieval is indistinguishable from uncompressed FP32 K/V at every
  context length tested.
- **Unlimited-context fails immediately at 1024 tokens.** Window=512;
  context 1024 means the needle (planted at ~10%, i.e. ~100 tokens in)
  rolled off the back of the cache after the first window closed. Bits
  jump from 0.01 to 25.46 — a 2²⁵ collapse in probability on the answer
  token. By 8192 tokens, all four engines have lost the answer (the
  model itself is the bottleneck, not the cache), and unlimited
  actually edges ahead by ~1.5 bits because its per-sub-window
  re-prefill gives the attention pattern a cleaner start.

The 0% top-1 across all four engines on all five lengths is a
**tokenisation-edge artifact**: Gemma's BPE splits "AURORA" into ` AUR` +
`ORA`. The model produces the correct first token (` AUR`) at 512–1024
tokens for the engines that preserve K/V, but the `needle_found` helper
does a literal substring search for "AURORA" in the decoded top-1 — which
fails on a partial-prefix token. **The bits column is the correct signal
here**; top-1 understates correctness when the answer is multi-token.

### 3. Conflict corpus exposes per-prompt steering differences

The aggregate (25% follow, 35% fallback) hides per-prompt churn.
**5 of 20 conflict prompts produced different verdicts** between
`standard` and `unlimited-context`:

| Prompt | standard | unlimited-context |
|---|---|---|
| `France → Lyon` | fallback (` Paris`) | other (` actually`) |
| `Mozart → Vienna` | fallback (` Salzburg`) | **follow (` Vienna`)** ← gained |
| `Gold → Gd` | follow (` Gd`) | other (` already`) ← lost |
| `Africa → Congo` | follow (` Congo`) | fallback (` Nile`) ← lost |
| `Mona Lisa → Picasso` | other (` Pablo`) | **follow (` Picasso`)** ← gained |

Net follow-rate is unchanged (5/20 either way), but the *which prompts*
shifted. The window-replay K/V reconstruction alters how the model
attends to the in-context premise: some overrides land better (Mozart,
Mona Lisa), some land worse (Gold, Africa), and the France prompt drops
into a non-answer ("In this fictional world… the capital of France is
actually [Lyon? Paris?]"). This is the engine-discriminating signal the
conflict corpus exists to surface — without it the engines look identical
on parametric, hide identical on needles (substring match misses), and
look identical on conflict aggregate.

`standard`, `markov-rs`, and `turbo-quant` all produced **identical
verdicts on all 20 conflict prompts**. The 4-bit codec on
`turbo-quant` did not alter any conflict outcome at this corpus size
on this model.

## What this baseline gates

| Regression | What would cause it |
|---|---|
| `Param %` < 69 for any engine | A K/V-state bug that changes the prefill-end hidden — almost any engine miscompile shows here first. |
| `Param bits` > 2.30 for any engine | Float drift large enough to shift token probabilities — likely a kernel regression. |
| `InCtx bits` > 23 for standard/markov-rs/turbo-quant on the 1024-tok needle | A regression in K/V preservation at <window length. |
| `InCtx bits` < 23 for unlimited-context on the 1024-tok needle | Either window=512 changed (config drift) or the engine started preserving K/V across windows. Either is suspicious without a corresponding code change. |
| Conflict verdicts on the same 5 divergence prompts change for standard | Standard is the K/V-baseline; its conflict outcomes should be tokeniser-deterministic. |

## Next runs to add to this baseline

- `gemma3-4b-2026-05-17-metal.json` — same suite with `--engines` running on Metal. Standard should be bit-identical to CPU; the question is whether the Metal-only engine impls preserve the same in-context bits.
- `gemma3-4b-32k-needles-2026-05-17.json` — extend `--needle-max-tokens 32768` to see whether `standard` also loses the needle at the long tail (likely yes around ctx ≈ 16K, since Gemma 3 4B's sliding-window attention is 4096 by default and the needle is past the window for the standard-attention layers).
- `gemma3-4b-with-apollo.json` — add `apollo` to `--engines`. Expected to perform poorly on parametric (uncalibrated injection point) but worth one data point.
- `mistral-7b-2026-05-17.json` — same suite against Mistral 7B. Different tokeniser, different RoPE, different sliding-window — should give a second cross-arch baseline.
