# CPU Decode Optimisation — Session Writeup (2026-05-15)

Single-session arc closing 41× of the gap to llama.cpp on Gemma 3 4B
Q4_K CPU decode. Machine: Apple M3 Max, 12 threads, BLAS / Accelerate
enabled, no GPU. Reference: llama.cpp build `6cd0cf72c` (7060) on the
same hardware.

## TL;DR

| Metric | Before | After | Δ |
|---|---:|---:|---:|
| Decode tok/s | 0.36 | **14.5–14.9** | **~40× faster** |
| Decode mean ms/tok | 2780 | 67–78 | ~37× lower |
| Gap to llama.cpp (decode) | 114× | **2.79×** | ~41× closed |
| Gap to llama.cpp (prefill) | ~55× | ~55× | unchanged |
| Unit-test coverage | 1255 passing | 1267 passing | +12 new tests |

All gains landed behind:
- Two `#[ignore]`d end-to-end parity tests on a real Q4_K vindex
  (`test_q4k_cached_parity`).
- Twelve new NEON-kernel parity tests against scalar references.
- `cargo fmt --check` clean.

## What the session did

Eight independent changes, each measurable in isolation, layered on
top of each other. Order matches the chronological progression; each
row's gain is conditional on the rows above it.

| Change | Decode | Δ vs prior | Δ vs baseline |

## What we tried that didn't work

Documented so future-us doesn't re-derive them:

- **Multi-superblock split-accumulator outer unroll**. Hypothesis:
  splitting the row's super-block loop into two parallel scalar
  accumulators lets the compiler interleave NEON work across
  super-blocks. Null result — LLVM was already producing that
  schedule via the inlined helper function. Reverted.
- **Fused gate+up matvec for x-cache locality**. Hypothesis: fusing
  two matvecs that share `h_in` keeps the 10 KB input hot in L1 across
  both weight decodes. The saving wasn't real: `h_in` is 10 KB, L1 is
  128 KB, and sequential matvecs already hit L1 on the second pass.
  Code kept (correctness-equivalent, one fewer Vec alloc per layer)
  but the perf hypothesis was wrong.
- **Fused K+V matvec** — same reason. Kept for symmetry.
- **`target-cpu=native` build flag**. Tried during a thermal-throttled
  run; no measurable difference from default release. The NEON
  intrinsics dominate; autovec on top is small.

## Where the remaining 2.79× lives

Bandwidth analysis on the same M3 Max:

| Engine | Q4_K bytes read / step | Decode ms | Effective bandwidth |
|---|---:|---:|---:|
| larql | ~2 GB | 67–70 | **~33 GB/s** |
| llama.cpp | ~2 GB | ~21 | **~95 GB/s** |
| LPDDR5 peak | — | — | ~400 GB/s |

llama.cpp is at ~24% of LPDDR5 peak; we're at ~8%. The 2.79× decode
gap matches the bandwidth-efficiency gap almost exactly. This is no
longer algorithmic territory — closing it is hand-tuned kernel work:

1. **Hand-rolled aarch64 asm** with explicit instruction interleaving
   and `prfm pldl1keep` prefetch. rustc/LLVM doesn't quite produce
   llama.cpp's hand-written schedule from intrinsics.
2. **Q3_K / Q2_K weights** — smaller bytes-per-weight, less memory
   traffic. Needs a new vindex extraction (separate workstream).

Open work itemised in `DIAGNOSIS.md` § "Remaining gap to llama.cpp".

## What about prefill?

55× behind llama.cpp on a 5-token prompt (2.6 s vs 47 ms). This
session didn't touch prefill — it still uses the legacy
`predict_q4k_prefill` path with full Q4_K → f32 dequant per layer
(~75 ms × 33 layers ≈ 2.5 s of dequant). The actual matmul work at
seq_len=5 is small; dequant is the dominant cost.

Closing prefill is the next round's headline. The fix is to route
prefill through a batched CPU `q4k_matmul` (multi-row matvec, weight
bytes read once and applied to all seq positions). Backend trait
declares `q4k_matmul` already; CPU backend returns `None`. Comparable
effort to the whole decode arc on this branch — and likely a bigger
absolute wallclock win, since prefill is ~37× larger than decode for
short prompts.

```
    --test test_q4k_cached_parity -- --ignored    # 2 e2e tests
cargo fmt --check                                  # clean
```

The two `#[ignore]`d integration tests load the real Gemma 3 4B Q4_K
vindex (~750 MB). They're skipped by default so CI doesn't need the
model, but both must pass before any merge that touches the decode
path:

- `cached_decode_matches_uncached_tokens` — bit-match required between
  the new cached prefill+decode and the legacy
  `predict_q4k_hidden`-per-step path. Catches any RoPE position drift,
  K/V append misorder, or PLE single-row regression.
- `direct_matvec_decode_matches_dequant_path` — direct-matvec decode
  must agree with the dequant decode on the first token (same
  arithmetic, same input from prefill) and allow up to one drift in
  the first three tokens (Q4_K matvec accumulates per super-block;
  dequant + sgemv accumulates per column; different summation orders
  produce small f32-rounding differences that compound over decode
  steps — both paths are mathematically correct).
